# 架構決策：六角形架構 + 雙專案 + SDL2-CS

日期：2026-06-26 14:10

## 核心訴求
遊戲核心邏輯需與多媒體輸出/輸入完全解耦，方便日後移植平台或更換多媒體方案，
且核心邏輯獨立不受影響。

## 定案

### 1. 架構：Ports & Adapters（六角形）
- **依賴方向單向：`App` → `Core`；`Core` 絕不認識 SDL。**
- `Core`（class lib，零 SDL）定義 **Ports（介面）**：
  - `IVideoBackend` — 畫面輸出（320×200 indexed 緩衝 + 調色盤）
  - `IAudioBackend` — 音訊輸出（已混好的 PCM；OPL 混音邏輯歸 Core）
  - `IInputBackend` — 輸入（鍵盤、搖桿）
  - `IClock` — 時間/延遲；`IFileSystem` — 資料檔/存檔
- `App`（WinExe）提供 **SDL Adapters**（`SdlVideo`/`SdlAudio`/`SdlInput`…）實作 Ports，
  進入點 `Program.cs` 為**組合根**注入。

### 2. 專案型態
- 單一 solution、**雙專案**：`Core`（class lib）+ `App`（WinExe），皆在 `cs_ported/`。
- 選雙專案而非單一專案：讓**編譯器強制隔離**——Core 不參考 SDL 組件，誤用即編譯失敗。
- 無 console 黑窗：`App` 設 `<OutputType>WinExe</OutputType>`。
- 視窗：SDL 開單一視窗，標題列右上 X 關閉，其餘 client 區即遊戲畫面。
- 建置輸出：單一 exe → `Build/`。

### 3. SDL C# 繫結：SDL2-CS
- 與原始 C 的 SDL2 呼叫 **1:1 對應**，移植幾近機械式照抄；體積最小（單檔 + SDL2.dll）。
- 音訊用 SDL Audio 輸出 + 移植過來的 OPL 軟體混音，不需 SDL_mixer。
- 替代方案備查：Silk.NET.SDL（較現代但 API 風格不同）、SDL3-CS（SDL3 與來源 SDL2 有落差）。

### 4. 網路多人
- **不納入**。移植時跳過 `network.c` / SDL2_net，以 stub 或條件略過。

## 結構草圖
```
cs_ported/
├─ AprCSTyrian.sln
├─ Core/        (class lib, 無 SDL) ── Ports + 遊戲邏輯
└─ App/         (WinExe, 參考 Core) ── SDL Adapters + Program.cs(組合根)
                                         輸出 → ../Build/
```
